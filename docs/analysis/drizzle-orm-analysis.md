# Drizzle ORM Analysis for `doc-repository-mcp`

**Audience:** Project owner / senior developer
**Status:** Technical evaluation — no code changes implied
**Date:** 2026-05-16

## 1. What Drizzle ORM is

Drizzle is a TypeScript-first SQL toolkit that sits deliberately closer to "typed query builder" than to "traditional ORM". Its design philosophy is summarized as *"if you know SQL, you know Drizzle"*.

Concretely:

- **Schema is TypeScript.** You declare tables with `pgTable("doc_pages", { id: uuid("id").primaryKey(), ... })`. The schema file is the single source of truth — types are *inferred* from it (`InferSelectModel`, `InferInsertModel`), not generated from a separate DSL file.
- **No code generation step for types.** Unlike Prisma, there is no `prisma generate` producing a client. Types exist the moment you write the schema. `drizzle-kit` exists only for *migration generation*, not for runtime client generation.
- **Two API layers**:
  - A relational query builder (`db.select().from(table).where(eq(...))`) that feels like LINQ / Kysely.
  - A raw `sql` tagged template (`sql\`SELECT ... FROM ${table}\``) that is *first-class*, not a fallback. You can interpolate Drizzle column references and table identifiers safely, and the result still flows through the driver.
- **No active record, no lazy loading, no proxy objects.** Results are plain objects. There is no `entity.save()`.
- **Driver-agnostic, but thin.** Drizzle does not ship its own connection pool or driver. It wraps an existing driver (in our case `postgres.js`) and translates query-builder calls into parameterized SQL that the driver executes.

### How it differs from Prisma / TypeORM

| Concern | Prisma | TypeORM | Drizzle |
|---|---|---|---|
| Schema source | `.prisma` DSL | TS decorators / entity classes | Plain TS (`pgTable`) |
| Runtime engine | Rust binary (query engine) | Pure JS | Pure JS, thin wrapper |
| Type generation | `prisma generate` step | tsc only | Inferred, no codegen |
| Migrations | Prisma Migrate (own format) | TypeORM migrations (JS) | Plain `.sql` files generated by `drizzle-kit` |
| Bun support | Historically problematic (Rust engine, query engine binary) | OK | First-class, advertised |
| Raw SQL ergonomics | `$queryRaw` — separate path, weak types | `query()` — untyped | `sql` template — typed, composable, idiomatic |
| pgvector / extensions | Awkward; requires `Unsupported` types or raw SQL escapes | Manual | Custom column types via `customType` helper |

The short version: Prisma owns the runtime; Drizzle gets out of the way.

## 2. Why it fits this stack specifically

Four properties of this project make Drizzle a particularly low-friction choice.

### 2.1 Same driver — `postgres.js`

`packages/db` already uses `postgres.js` v3. Drizzle's `drizzle-orm/postgres-js` adapter wraps exactly that driver. There is no driver swap, no pool reconfiguration, no change to connection-string handling, SSL options, or `prepare: false` workarounds. The existing `sql` instance can be handed to Drizzle:

```ts
import postgres from "postgres";
import { drizzle } from "drizzle-orm/postgres-js";

const client = postgres(DATABASE_URL); // existing client
export const db = drizzle(client, { schema });
```

Both `db` (Drizzle) and `client` (raw `sql\`...\``) can coexist on the same connection pool, which is critical for incremental migration.

### 2.2 No native binary

Prisma ships a Rust query engine. On Bun + Linux Alpine (Dockerfile), this has historically been the single largest source of "works on my machine" failures: glibc vs. musl binaries, Bun's incomplete N-API surface, slow cold starts in containers. Drizzle has zero native dependencies — it is pure TypeScript compiled to JavaScript. The Docker image stays small, cold start stays fast, and there is no `engineType` decision to make.

### 2.3 First-class Bun support

Drizzle explicitly tests against Bun and documents `bun:sqlite`, `drizzle-orm/bun-sql`, and `postgres-js` paths. There are no known Bun-specific footguns. `drizzle-kit` itself runs under `bun x drizzle-kit ...` without issue.

### 2.4 TypeScript-first

The codebase is already aggressively typed (Zod contracts in `packages/contracts`, `packages/admin-contracts`). Drizzle's inferred row types compose naturally with Zod — you can derive a Zod schema from a Drizzle table via `drizzle-zod`, eliminating one class of drift between DB rows and DTOs.

## 3. pgvector support

This is the load-bearing question for this project. The honest answer: **Drizzle handles the storage side cleanly, the query side partially.**

### 3.1 Column type

Drizzle ships a built-in `vector` column type in `drizzle-orm/pg-core` (added in 0.31+):

```ts
import { pgTable, uuid, vector, index } from "drizzle-orm/pg-core";

export const docEmbeddings = pgTable(
  "doc_embeddings",
  {
    id: uuid("id").primaryKey(),
    chunkId: uuid("chunk_id").notNull(),
    embedding: vector("embedding", { dimensions: 1536 }),
  },
  (table) => ({
    embeddingHnsw: index("doc_embeddings_embedding_hnsw")
      .using("hnsw", table.embedding.op("vector_cosine_ops"))
      .with({ m: 16, ef_construction: 64 }),
  }),
);
```

The `vector` type, HNSW `using(...)`, and operator class (`vector_cosine_ops`) are all first-class. `drizzle-kit generate` emits the correct `CREATE INDEX ... USING hnsw (embedding vector_cosine_ops)` DDL.

What still requires manual SQL in the migration file:

- `CREATE EXTENSION IF NOT EXISTS vector;` — Drizzle does not manage extensions. You put this in the first migration by hand (one line).
- Index tuning parameters beyond `m` / `ef_construction` are passed via `.with({...})` and round-trip correctly.

### 3.2 Similarity operators in queries

This is where Drizzle's abstraction stops being magical. pgvector's operators (`<=>` cosine distance, `<->` L2, `<#>` inner product) are not first-class in Drizzle's query builder. There are two viable patterns:

**Pattern A — `sql` interpolation (recommended):**

```ts
import { sql } from "drizzle-orm";

const rows = await db
  .select({
    chunkId: docEmbeddings.chunkId,
    distance: sql<number>`${docEmbeddings.embedding} <=> ${queryVec}::vector`.as("distance"),
  })
  .from(docEmbeddings)
  .orderBy(sql`${docEmbeddings.embedding} <=> ${queryVec}::vector`)
  .limit(k);
```

You get typed table refs, typed projection, parameterized binding of `queryVec`, and the result is a typed `{ chunkId: string; distance: number }[]`. The `<=>` is just a SQL literal — that is fine.

**Pattern B — `customType` with helper functions.** You can wrap `cosineDistance(col, vec)` in a helper that returns `sql`. This is cosmetic.

### 3.3 Hybrid keyword + vector retrieval

The current `RemoteDocsStorage` builds queries that combine `ts_rank`, `vector` distance, optional source filters, and limits. These will remain `sql` tagged templates in Drizzle. That is not a regression — Drizzle's `sql` is as ergonomic as `postgres.js`'s, with the bonus of typed column references (`${docChunks.body}` instead of stringly-typed `"doc_chunks"."body"`).

## 4. Migration story

### 4.1 `drizzle-kit` flow

```
schema.ts  ──drizzle-kit generate──▶  migrations/0001_xxx.sql  ──drizzle-kit migrate──▶  PostgreSQL
                                       (plain SQL, committed)         (tracked in __drizzle_migrations)
```

- **Generated files are plain SQL.** Compatible with the project's current "SQL files on disk" approach. They are diff-friendly, code-reviewable, and can be hand-edited before being applied (essential for things like `CREATE EXTENSION`, custom DDL, data backfills).
- **Migration tracking.** Drizzle records applied migrations in `__drizzle_migrations` (schema configurable). The current custom runner has *no* tracking table — this is an upgrade. Drizzle uses content hashes, so renaming a migration file does not cause re-runs.
- **`drizzle-kit push`** exists for prototyping (diff schema → DB directly, no migration files) but should **not** be used here. Stick to `generate` + `migrate` for auditability.

### 4.2 Coexistence with existing migrations

There are two ways to onboard:

1. **Baseline.** Treat the current DB state as canonical. Write the Drizzle schema to match. Generate an empty initial migration and *mark it applied* by inserting into `__drizzle_migrations` directly. New migrations go through Drizzle only. The existing `migrations/remote-docs/*.sql` directory is retired (or kept read-only for history).
2. **Re-baseline.** Drop the custom runner, copy the existing `.sql` files into Drizzle's migrations directory, let Drizzle apply them on a fresh DB. Existing databases get a one-time `__drizzle_migrations` seed script.

Option 1 is lower risk for production; option 2 is cleaner long-term. Recommend option 1.

## 5. What it replaces

The biggest quality-of-life win is eliminating the manual row interface + mapper layer.

**Before:**
```ts
interface PageRow {
  id: string;
  source_id: string;
  url: string;
  title: string | null;
  fetched_at: Date;
}

function mapPage(row: PageRow): DocPage {
  return {
    id: row.id,
    sourceId: row.source_id,
    url: row.url,
    title: row.title,
    fetchedAt: row.fetched_at,
  };
}

const rows = await sql<PageRow[]>`SELECT id, source_id, url, title, fetched_at FROM doc_pages WHERE source_id = ${sourceId}`;
return rows.map(mapPage);
```

**After:**
```ts
const rows = await db.select().from(docPages).where(eq(docPages.sourceId, sourceId));
// rows: { id: string; sourceId: string; url: string; title: string | null; fetchedAt: Date }[]
```

Drizzle column names already use camelCase JS keys mapped to snake_case DB columns. The mapper is gone. The row interface is gone. The SELECT list maintenance is gone.

For ~9 tables with ~6 columns average, that is roughly 9 × (interface + mapper) = ~18 boilerplate blocks deleted.

## 6. What it does NOT replace

Be honest about the boundary:

- **Hybrid retrieval queries** (keyword + vector + filter + rank) stay in `sql` templates. Drizzle does not make these meaningfully shorter; it only makes the column references typed.
- **pgvector operators** (`<=>`, `<->`, `<#>`) are not abstracted. You write them as SQL literals inside `sql`.
- **HNSW query-time parameters** (`SET LOCAL hnsw.ef_search = 100;`) are session-level pragmas — same `sql.unsafe()` story as today.
- **Full-text search** (`tsvector`, `to_tsquery`, `ts_rank_cd`) is partially supported (there is a `tsvector` custom-type pattern) but ranking expressions stay as `sql`.
- **Migrations involving data backfill** still need hand-written SQL inside the generated migration file. Drizzle does not auto-generate data transformations.
- **The auth / admin tables** (`admin_users`, `admin_sessions`, `admin_audit_events`) are pure CRUD — these benefit fully. The docs tables are mixed.

Rough estimate: ~70% of current queries (simple SELECT/INSERT/UPDATE) move to the query builder; ~30% (similarity search, hybrid retrieval, refresh-job state machines) stay as `sql` templates with typed column refs.

## 7. Trade-offs

### Gains

- Elimination of row-interface + mapper boilerplate (~9 tables worth).
- Compile-time safety on column names — renaming a column is now a tsc error everywhere it is referenced.
- Real migration history tracking (current runner has none).
- `drizzle-kit studio` — a free local DB browser, useful in development.
- `drizzle-zod` — DTOs can be derived from schema, reducing drift with `packages/contracts`.
- Easier onboarding for new developers — the schema lives in one TS file, not scattered across `.sql` files.

### Costs

- **Refactor scope.** `RemoteDocsStorage` is the main rewrite target. Roughly 30–50 methods to migrate. Mechanical but not trivial — ~2–4 days of focused work.
- **Migration baseline carry.** One-time chore to seed `__drizzle_migrations` on existing environments.
- **Learning curve.** Small but real — operators (`eq`, `and`, `inArray`, `gte`), ordering, joins, partial selects. A senior dev picks it up in a day.
- **Risk surface.** Any rewrite of DB code risks subtle behavior changes (NULL semantics, boolean coercion, date-time precision). Mitigated by good tests, but the tests must exist first.
- **New dependency.** `drizzle-orm` + `drizzle-kit` (dev). Both pure JS, no native deps. Maintenance burden is low but non-zero.

### Non-issues (often raised but not real here)

- "ORMs are slow" — Drizzle compiles to the same SQL `postgres.js` would have run. No measurable runtime overhead.
- "Lock-in" — schema is plain TS, migrations are plain SQL. Walking away later means deleting `drizzle-orm` and writing mapper functions again.

---

## 8. Verdict

**Recommended, with one condition: do it before `RemoteDocsStorage` doubles in size.**

The case for Drizzle in this project is unusually strong because of the driver alignment (`postgres.js`), the Bun runtime constraint (no Rust engine), and the existing pain (manual row mappers, no migration tracking). The case against is the standard rewrite-cost argument, which is bounded here because the storage layer is centralized in `packages/db`.

**Adopt if:**
- The storage layer is going to keep growing (more entities, more admin features). The boilerplate compounds.
- Migration history tracking is starting to matter (multi-environment deploys, rollbacks).
- The team wants `drizzle-zod` to unify DB row types with contract DTOs.

**Defer if:**
- The schema is about to undergo a major redesign — wait, then model the new shape in Drizzle once.
- The next 1–2 sprints are entirely retrieval-quality work (hybrid search, reranking) — that work stays in `sql` either way, so Drizzle adds no value to *that* sprint.

**Do not adopt if:**
- The intent is to abstract away pgvector. It will not. Use a vector DB if that is the goal.

Net: this is the right tool for this stack. The migration is bounded, mechanical, and pays for itself within the next 5–10 storage-layer changes.
